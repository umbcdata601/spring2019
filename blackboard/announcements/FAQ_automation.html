Question: "How does Python read and interpret a response?"
Response: One reason Jupyter can be complicated to learn is because it is composed of many different tools stacked in layers. Understanding how you are able to write code on a computer relies on being able to grasp each tool and how it relates to other tools in the stack. For a quick overview, see How Jupyter works. As indicated in the figure on that page, you rely on a web browser (eg Firefox, Chrome, Internet Explorer) to access a webserver running on your local computer. The server manages your interaction with both a .ipynb file and the kernel being used. We use a Python kernel to execute code written in the browser.

The Python kernel is effectively serving in Jupyter as the REPL. Addressing how that works is the content of this series of posts.

Question: "How to automate sending encrypted emails?"
Response: There are two ways to interpret this question. One aspect is the encryption between the sender (you) and the server (eg Gmail), the other is encrypting the email body so that no one besides the intended recipient can view the content.

When you use SMTP Secure, the mail is transmitted in a encrypted connection but the owner of the mail server could potentially read the content. (This is how Gmail filters out spam and monetizes email with advertisements -- by reading the content).

Encrpting the body of the email before sending it to the server is usually a separate step. PGP is a standard for encryption (and applies to more than just email; see the Wikipedia entry). In this context, there are libraries in Python that work with PGP.

Question: "Automate booking tickets online. Ping me on my phone when movie tickets are released."
Response: This question can be tackled by considering how to break down a seemingly straight-forward objective into a sequence of smaller tasks.

For example, getting a message to your phone means changing "ping me on my phone" to "send a text message to a phone number" which I demonstrated in the class on automation.

Checking whether an API for ticket services exists is worth doing, but the task of "booking tickets online" implies a likely web interaction since an API typically is not offered. Then the question becomes how to interact with a webpage. Selenium is a candidate.

Lastly, a recurring task needs to be configured to run the check of ticket availability. This means having a dedicated computer available (either at home or in the cloud) from which to run a recurring task.
